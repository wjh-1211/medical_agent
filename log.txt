T012 RAG 链路改造说明
======================

1. 这次要解决什么问题
----------------------

T011 之前，Agent 已经可以记住用户事实和会话摘要，但它不能从受控外部文档中获取事实。

例如：

  用户问：受控知识库中，出现呼吸困难时有什么提示？请给出来源。

旧链路只能依赖 DeepSeek-R1 本身已经学到的参数化知识。模型可能回答得看似合理，但无法证明答案来自哪个文档，也可能虚构“医学指南”或“相关文献”。

这次 RAG 改造的目标是：

  受控知识文档
  -> 切分为可检索 chunk
  -> 独立 Embedding 模型生成向量
  -> 向量检索返回证据
  -> 证据进入 ReAct Observation
  -> 主模型根据证据回答，并附真实来源

注意：

  Memory 保存“这个用户的信息”。
  RAG Knowledge 保存“外部受控文档的事实”。

两者不混存。


2. 新增了哪些组成部分
------------------------

2.1 LangChain4j

在 pom.xml 中引入 LangChain4j 1.1.0。

本项目没有把 AgentKernel 替换成 LangChain4j Agent Executor。

LangChain4j 当前承担的是 RAG 基础设施职责：

  - DocumentSplitters：将文档切分为 chunk
  - EmbeddingModel：定义向量模型接口
  - TextSegment / Document：承载文档与 chunk 元数据

现有 AgentKernel、Skill、ToolRouter 和 ReAct 仍是项目自己的编排骨架。


2.2 受控知识文档

新增：

  knowledge/medical-rag-demo.json

该文件是本地演示语料，内容包含：

  - 呼吸困难、说话困难、嘴唇或面色发青、意识模糊时应尽快寻求紧急医疗帮助
  - 发热伴症状持续加重、胸痛、持续高热等线下评估提示
  - 知识库未命中时不得伪造来源

每个文档 section 都会获得以下可追溯元数据：

  chunkId
  content
  source
  section
  documentVersion
  contentHash


2.3 独立 Embedding 模型

生成回答仍使用：

  /mnt/Data/multi-agent/DeepSeek-R1-Distill-Qwen-7B

向量化改为使用独立模型：

  /mnt/Data/multi-agent/Qwen/Qwen3-Embedding-4B

新增脚本：

  scripts/local_embedding_inference.py

它以常驻 Python 进程加载 Qwen3-Embedding-4B，接收 document 或 query 文本，返回归一化向量。

为什么不能直接用 DeepSeek-R1 做 Embedding？

  DeepSeek-R1 是生成模型，职责是组织语言和做推理。
  Qwen3-Embedding-4B 是针对文本相似度与检索训练的模型。
  将两者分开，才能让“生成”和“检索”各自使用合适模型。


2.4 SQLite 向量索引

新增：

  SqliteKnowledgeVectorStore

索引数据库默认位置：

  data/knowledge-index.db

每次本地 CLI 启动时：

  1. KnowledgeDocumentLoader 读取 knowledge 目录下的 JSON 或 Markdown。
  2. LangChain4j DocumentSplitters 切分文档。
  3. Qwen3-Embedding-4B 为每个 chunk 生成向量。
  4. chunk、元数据、向量写入 SQLite。

查询时：

  1. Qwen3-Embedding-4B 为 query 生成向量。
  2. SQLite 读取当前受控知识 chunk。
  3. SqliteKnowledgeVectorStore 计算余弦相似度。
  4. 返回 Top-K 证据。

当前使用 SQLite 是有意的：

  - 演示语料小，方便离线运行和审阅。
  - 不需要额外启动 Qdrant 服务。
  - 当前实现是全量向量扫描，适合小语料。

未来知识库变大后，可替换为 Qdrant。下一步先在 T013 建立质量基线与报告，再在 T014 增加 BM25、RRF 和 rerank。


2.5 KnowledgeSearchSkill

新增 Tool：

  knowledge_search

输入示例：

  {
    "query": "呼吸困难 就医提示",
    "topK": 3
  }

输出示例：

  {
    "status": "ok",
    "query": "呼吸困难 就医提示",
    "found": true,
    "chunks": [
      {
        "chunkId": "5af167fe3dddf25ebe902074",
        "content": "出现呼吸困难、说话困难、嘴唇或面色发青、意识模糊时，应尽快寻求紧急医疗帮助。",
        "source": "受控医疗知识演示：呼吸道症状就医提示",
        "section": "呼吸困难的紧急提示",
        "documentVersion": "2026-07-demo",
        "score": 0.91
      }
    ]
  }

KnowledgeSearchSkill 只负责检索和返回 JSON。

它不负责：

  - 医疗诊断
  - 用药建议
  - Prompt 拼接
  - 自己调用其他 Skill
  - 自己生成自然语言答案


3. 一次查询如何经过整个 Agent 链路
--------------------------------------

示例用户输入：

  受控知识库中，出现呼吸困难时有什么提示？请给出来源。

完整运行顺序：

  User
    -> AgentCliRunner
    -> AgentContextFactory
    -> AgentKernel
    -> Session / Long-term / Summary Memory Recall
    -> KnowledgeRetrievalDecisionAgent
    -> knowledge_search Tool
    -> KnowledgeSearchSkill
    -> Qwen3-Embedding-4B Query Embedding
    -> SQLite Vector Search
    -> JSON Observation
    -> 主 ReAct Prompt
    -> DeepSeek-R1 Final Answer
    -> Source Marker Serialization
    -> Session / Summary / Long-term Memory Update
    -> CLI 输出

逐步解释：

步骤 1：CLI 组装请求

  AgentCliRunner 将当前用户消息、sessionId、userId 和 history 放进 AgentRequest。

步骤 2：构造统一上下文

  AgentContextFactory 将 AgentRequest 转换为 AgentContext。

步骤 3：先召回三层 Memory

  AgentKernel 先通过 ToolRouter 尝试读取：

    session_memory_read
    long_term_memory_read
    summary_memory_read

  这些是用户相关信息，不是 RAG 文档。

步骤 4：LLM 判断是否需要外部证据

  KnowledgeRetrievalDecisionAgent 使用独立的短 Prompt。

  当用户明确提到“受控知识库”、来源、指南、文档证据或引用时，
  该 Agent 应输出：

    {
      "type": "tool_call",
      "toolName": "knowledge_search",
      "input": {
        "query": "呼吸困难 就医提示",
        "topK": 3
      }
    }

  这是 LLM 的语义决策。
  Java 没有写“如果消息包含知识库就检索”的关键词 if-else。

步骤 5：Router 执行 Skill

  ToolRouter 根据 toolName 找到 KnowledgeSearchSkill。

  KnowledgeSearchSkill 调用向量检索，返回带来源的 JSON。

步骤 6：检索结果进入 Observation

  AgentKernel 将 Tool 输入和 Tool 输出包装成 Observation。

  下一次主 ReAct Prompt 会看到：

    toolName = knowledge_search
    result.chunks[0].content
    result.chunks[0].source
    result.chunks[0].chunkId

步骤 7：DeepSeek-R1 组织最终回答

  主模型根据 Observation 生成对用户可见的中文回答。

  Prompt 要求它不伪造指南、文献、source 或 chunkId。

步骤 8：追加真实来源标记

  如果本轮确实有 knowledge_search Observation，AgentKernel 会从 Tool Result 中取回真实 source 与 chunkId，并追加：

    [source: 受控医疗知识演示：呼吸道症状就医提示 | chunk: 5af167fe3dddf25ebe902074]

  这个步骤只做来源字段的确定性序列化。
  它不判断模型的每一句话是否真的被证据支持。
  “逐句事实是否被证据支持”属于后续 Grounding Check。


4. 真实 CLI 验收例子
---------------------

启动：

  ./cli.sh local

输入：

  受控知识库中，出现呼吸困难时有什么提示？请给出来源。

本次真实运行的输出：

  受控知识库中提到，出现呼吸困难、说话困难、嘴唇或面色发青、意识模糊时，应尽快寻求紧急医疗帮助。知识库提供的示例包括相关症状的就医提示，如发热伴症状持续加重、明显胸痛、持续高热或基础疾病时需及时就医。 [source: 受控医疗知识演示：呼吸道症状就医提示 | chunk: 5af167fe3dddf25ebe902074]

这说明：

  - 本地文档被向量化。
  - 模型触发了 knowledge_search。
  - 检索结果进入了 ReAct。
  - 回答使用了受控文档中的事实。
  - 来源标记来自真实 Tool Result。

6. 当前不是完整 RAG 的部分
----------------------------

当前已经是标准 Vector RAG MVP，但还没有：

  - BM25 关键词检索
  - Hybrid Retrieval
  - RRF 融合
  - Reranker
  - Query Rewrite
  - 多跳检索
  - Qdrant 等大规模向量数据库
  - Recall@K、MRR、nDCG 等检索准确率指标
  - Grounding Check
  - 医疗 Guardrail

因此当前可以准确描述为：

  使用 LangChain4j、Qwen3-Embedding-4B、SQLite 与 Tool Calling 实现的本地 Agentic Vector RAG 最小闭环。

不应描述为：

  企业级 Hybrid RAG 或已经完成医疗事实可信校验的系统。
