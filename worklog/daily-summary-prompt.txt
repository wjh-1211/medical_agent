你现在要为本项目生成“每日工作总结”，但这不是普通日报。

你的目标是：

1. 帮助开发者快速回顾今天到底推进了什么。
2. 强调每项修改背后的“为什么”，而不是只罗列文件名。
3. 说明这些改动“怎么用”，让第二天接手的人可以直接继续推进。
4. 明确这些改动对当前 Task、Sprint 和后续架构的意义。

请严格按照以下要求输出。

一、总结原则

- 不要写空话，不要写“完成了若干工作”“进行了优化”这类无信息表达。
- 优先回答三个问题：
  - 今天为什么要做这些改动？
  - 这些改动具体怎么用？
  - 这些改动为后续哪一步铺路？
- 如果当天没有完成某项工作，要明确写出卡点和原因。
- 输出必须面向项目协作，而不是面向个人情绪记录。

二、输出结构

请按下面结构输出：

【今日目标】
- 今天所属的 Sprint / Task
- 今天原本要解决的问题

【实际完成】
- 每一项完成内容都用下面格式写：
  - 改动项：
  - 涉及文件/目录：
  - 为什么这样做：
  - 怎么使用：
  - 对后续的意义：

【关键设计决策】
- 只写真正影响后续开发的设计决策
- 每项说明：
  - 决策内容：
  - 不这样做会有什么问题：
  - 后续开发需要遵守什么约束：

【未完成与阻塞】
- 尚未完成什么
- 为什么没完成
- 下一步准备怎么解决

【明日建议】
- 明天最应该优先推进的 1 到 3 件事
- 每件事说明为什么优先

【文件形式】
- 每日增量更新写入，不要覆盖之前的内容，非全删全写模式

三、输出风格

- 用中文
- 简洁、专业、可执行
- 不写口号式总结
- 不写泛泛表扬
- 不要只按文件罗列，必须突出“目的”和“使用方式”

四、如果我给你的是代码改动或 Git diff

你要先自行归纳这些改动属于哪几个主题，再输出总结。

五、如果我给你的是当天开发过程描述

你要主动提炼：

- 当前 Task 的推进程度
- 关键架构选择
- 哪些内容已经形成可复用资产
- 哪些内容只是临时实现

六、最终输出要求

最终总结必须让我看完之后能直接知道：

- 今天推进了什么
- 为什么这样推进
- 这些成果明天怎么继续用
- 还有哪些风险没有解决

如果信息不足，请先基于已有信息给出结构化总结，再列出缺失信息，不要停止输出。
